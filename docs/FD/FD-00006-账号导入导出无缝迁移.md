## 一、功能概述

### 1.1 设计目标

在现有导入导出功能基础上，实现 **"导出一个文件 → 不做任何修改 → 直接导回来"** 的无缝账号迁移能力：

1. **导入端**：支持混合类型账号文件的一键智能导入（自动识别 Outlook / 标准 IMAP / 自定义 IMAP / GPTMail）。
2. **导入端**：支持邮箱域名自动推断 provider、自动按 provider 分组、重复账号处理策略。
3. **导出端**：增强导出文件（头部元信息 + GPTMail 临时邮箱纳入导出）。
4. **完全向后兼容**：现有单类型导入（选择特定 provider）行为不变。

### 1.2 核心约束

1. **不改变现有 API URL 与基本响应结构**：`POST /api/accounts` 通过新增 `provider="auto"` 值触发智能识别，不影响原有 provider 值。
2. **不改变数据库 schema**：无需新增表或列，复用现有 `accounts`、`groups` 表结构。
3. **不改变导出安全机制**：2FA 验证 Token（IP + User-Agent 绑定）保持不变。
4. **不改变现有单类型导入逻辑**：`provider != "auto"` 时走原有代码路径。

### 1.3 Out of Scope

- JSON / CSV / Excel 格式导入导出（后续版本）
- 导出文件加密（已有 2FA 保护导出操作）
- 跨网络的实时同步迁移

**辅助函数**：

```python
def infer_provider_from_email(email: str) -> Optional[str]:
    """
    从邮箱地址推断 provider。
    返回 provider key（如 'gmail'、'qq'）或 None（无法推断）。
    """
    if not email or "@" not in email:
        return None
    domain = email.rsplit("@", 1)[-1].strip().lower()
    return DOMAIN_PROVIDER_MAP.get(domain)
```

### 4.2 智能识别核心函数

**新增函数**：`_detect_line_type(line, parts, fallback_host, fallback_port)`

```python
def _detect_line_type(
    line: str,
    parts: list,
    fallback_host: str = "",
    fallback_port: int = 993,
) -> Dict[str, Any]:
    """
    根据分隔后的段数和内容特征，自动判断一行账号的类型。

    返回:
        {
            "type": "outlook" | "imap" | "gptmail" | "error",
            "provider": str,          # 具体 provider key
            "fields": dict,           # 解析出的字段
            "error": str | None,      # 错误信息（type=="error" 时）
            "auto_group_name": str,   # 自动分组名
        }
    """
```

**识别流程（按优先级从高到低）：**

```
输入: line（去除首尾空白后）
步骤 0: 跳过空行和注释行（以 # 开头）

步骤 1: 按 "----" 分割
      parts = line.split("----")
      n = len(parts)

步骤 2: 段数判断

  ┌─ n >= 5 且 parts[2].strip().lower() == "custom"
  │   → 自定义 IMAP
  │   → fields: {email, imap_password, imap_host, imap_port}
  │   → provider: "custom"
  │
  ├─ n >= 4（且不满足上条）
  │   → Outlook（OAuth）
  │   → fields: {email, password, client_id, refresh_token}
  │   → refresh_token = "----".join(parts[3:])  # token 中可能含 "----"
  │   → provider: "outlook"
  │
  ├─ n == 3
  │   → 第3段是 provider 名？
  │   ├─ 是已知 provider → 标准 IMAP（带显式 provider）
  │   │   → fields: {email, imap_password, provider}
  │   │   → 从 MAIL_PROVIDERS 获取 host/port
  │   └─ 不是 → 报错（未知的 3 段格式）
  │
  ├─ n == 2
  │   → 域名推断 provider
  │   ├─ 推断成功 → 标准 IMAP
  │   │   → fields: {email, imap_password}
  │   │   → 从 MAIL_PROVIDERS 获取 host/port
  │   │   ★ 特殊处理：微软域名见 4.3 节
  │   └─ 推断失败 → 自定义 IMAP（兜底）
  │       → fields: {email, imap_password, imap_host: fallback_host, imap_port: fallback_port}
  │       → 如果 fallback_host 也为空 → 报错
  │
  └─ n == 1
      → 检查是否是合法邮箱地址
      ├─ 是 → GPTMail 临时邮箱
      │   → fields: {email}
      │   → provider: "gptmail"
      └─ 否 → 报错（无法解析）
```

### 4.3 微软域名在 2 段格式下的特殊处理

当 2 段格式的邮箱域名为 `@outlook.com` / `@hotmail.com` / `@live.com` 时，存在歧义：

- Outlook OAuth 账号必须有 4 段（含 client_id + refresh_token），2 段不可能是 OAuth。
- 因此 **2 段微软域名一律按 IMAP 密码登录处理**。

处理方式：
- `account_type = "imap"`
- `provider = "outlook"`（复用现有 outlook provider 的 IMAP 配置：`outlook.office365.com:993`）
- 但注意：微软 IMAP 通常需要 XOAUTH2 认证，**普通密码可能无法登录**。建议在该行的导入结果中附加警告：`"⚠ 微软邮箱仅 2 段格式，可能需要 OAuth 认证（4段格式）"`

### 4.4 混合导入主函数

**新增函数**：`_handle_auto_import(data, request)`

```python
def _handle_auto_import(data: dict, request) -> tuple:
    """
    处理 provider="auto" 的智能混合导入。

    参数:
        data: 请求 JSON，包含:
            - account_string: str       # 多行文本
            - duplicate_strategy: str   # "skip"(默认) | "overwrite"
            - imap_host: str            # 兜底 IMAP host（可选）
            - imap_port: int            # 兜底 IMAP port（可选）
        request: Flask request 对象

    返回:
        Flask response (JSON)
    """
```

**处理流程：**

```
1. 解析参数
   - account_string → 按换行拆分为行列表
   - duplicate_strategy: "skip" | "overwrite"（默认 "skip"）
   - fallback_host, fallback_port: 兜底 IMAP 设置
   - explicit_group_id: data.get("group_id")  # 可能为 None 或具体 ID

2. 判断分组模式
   # ★ 设计决策：provider=auto 时，group_id 的处理策略
   #   - group_id 为 None → 自动按 provider 分组（主流程）
   #   - group_id 为具体数值 → 所有账号统一使用该分组（用户通过 API 指定）
   use_auto_group = (explicit_group_id is None)

3. 初始化计数器
   - imported = 0, skipped = 0, failed = 0
   - by_provider = {}  # 按 provider 分类统计
   - groups_created = []  # 新创建的分组列表
   - errors = []
   - group_cache = {}  # provider → group_id 缓存（仅 use_auto_group=True 时使用）

4. 逐行处理
   for line_num, line in enumerate(lines, 1):
       # 跳过空行和注释
       if not line.strip() or line.strip().startswith('#'):
           continue

       # a. 检测类型
       result = _detect_line_type(line, parts, fallback_host, fallback_port)
       if result["type"] == "error":
           errors.append({line, email, error})
           failed += 1
           continue

       # b. 解析并校验字段
       fields = result["fields"]
       email = sanitize_input(fields["email"])
       # 校验邮箱格式...

       # c. 解析分组
       provider = result["provider"]
       if use_auto_group:
           group_id = _resolve_auto_group(provider, group_cache, groups_created)
       else:
           group_id = explicit_group_id

       # d. 处理重复
       existing = accounts_repo.get_account_by_email(email)
       if existing:
           if duplicate_strategy == "skip":
               skipped += 1
               continue
           elif duplicate_strategy == "overwrite":
               _overwrite_account(existing, result, group_id)
               imported += 1
               continue

       # e. 新增账号
       if result["type"] == "outlook":
           accounts_repo.add_account(
               email=email,
               password=fields["password"],
               client_id=fields["client_id"],
               refresh_token=fields["refresh_token"],
               group_id=group_id,
               account_type="outlook",
               provider="outlook",
           )
       elif result["type"] == "imap":
           accounts_repo.add_account(
               email=email,
               password="",
               client_id="",
               refresh_token="",
               group_id=group_id,
               account_type="imap",
               provider=provider,
               imap_host=fields.get("imap_host", ""),
               imap_port=fields.get("imap_port", 993),
               imap_password=fields.get("imap_password", ""),
           )
       elif result["type"] == "gptmail":
           _handle_gptmail_import(email, group_id, ...)

       imported += 1
       by_provider[provider] = by_provider.get(provider, {})
       by_provider[provider]["imported"] = by_provider[provider].get("imported", 0) + 1

4. 提交事务 & 返回结果
```

```python
def _resolve_auto_group(
    provider: str,
    group_cache: dict,
```

```python
def _handle_gptmail_import(email: str, group_id: int, errors: list, line_num: int) -> bool:
    """
    处理 GPTMail 临时邮箱的导入。

    流程:
        1. 检查邮箱是否已在系统中
        2. 调用 GPTMail API 验证邮箱是否可用
           - gptmail.get_temp_emails_from_api(email)
           - 如果返回成功 → 邮箱可用

           - 从邮箱地址提取 prefix 和 domain
           - gptmail.generate_temp_email(prefix, domain)
        4. 成功 → 添加到 accounts 表（account_type='imap', provider='gptmail'）
        5. 失败 → 记录到 errors 列表

    注意:
        - GPTMail 账号在 accounts 表中存储为:
          account_type='imap', provider='gptmail'
          client_id='', refresh_token='', imap_password=''
          imap_host 和 imap_port 留空（通过 API 访问，不走 IMAP）
        - 需要做 API 调用频率限制，避免批量导入时触发 GPTMail API 限流
    """
```

## 九、错误处理与边界情况

### 9.1 错误分类

| 错误类型 | 触发条件 | 处理方式 |
|----------|----------|----------|
| 格式错误 | 行无法匹配任何已知格式 | 记入 errors 列表，继续处理下一行 |
| 邮箱格式无效 | 正则校验失败 | 记入 errors 列表 |
| 未知域名无兜底 | 2段 + 未知域名 + 无 fallback host | 记入 errors 列表 |
| GPTMail API 失败 | API 调用超时/返回错误 | 记入 errors 列表 |
| 数据库错误 | INSERT/UPDATE 异常 | 回滚当前行，记入 errors |
| 3段未知 provider | 第3段不在已知 provider 列表中 | 记入 errors 列表 |

### 9.2 事务策略

- **混合导入采用逐行提交**（而非整体事务），确保部分失败不影响已成功的行。
- 理由：混合文件可能包含数百行不同类型账号，整体回滚用户体验差。
- 通过 `imported / skipped / failed` 计数让用户清楚知道每行的处理结果。

### 9.3 性能考量

| 场景 | 策略 |
|------|------|
| 大批量导入（1000+ 行） | 分组查找使用 `group_cache` 字典缓存，避免重复 DB 查询 |
| 重复检查 | `get_account_by_email` 逐行查询；可考虑批量预加载优化 |
| GPTMail API 调用 | 批量导入时做简单限流（每行间隔 100ms），避免 API 429 |
| 导出大量账号 | `_build_export_text` 使用 `io.StringIO` 拼接，避免频繁字符串连接 |

### 9.4 安全注意事项

| 风险 | 缓解 |
|------|------|
| 导出文件含明文凭据 | 保持 2FA 验证不变；建议用户导出后妥善保管 |
| 覆盖更新可能破坏已有账号 | 审计日志记录所有覆盖操作；UI 上有明确警告 |
| 恶意文件注入 | 复用现有 `sanitize_input()` 对每个字段做 XSS 防护 |
| 批量 GPTMail 注册滥用 | 限制单次导入的 GPTMail 账号数量（建议上限 20） |
