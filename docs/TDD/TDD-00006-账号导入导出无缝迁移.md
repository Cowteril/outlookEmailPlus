## 1. 文档目的

本 TDD 描述 PRD-00006「账号导入导出无缝迁移」的**完整技术实现细节**，重点回答：

- 如何在**不新增 API URL** 的前提下，通过 `POST /api/accounts` 的 `provider="auto"` 实现混合导入
- 如何实现**一份导出文件可直接导回**（含 Outlook / 标准 IMAP / 自定义 IMAP / GPTMail）
- 如何在导入时实现**域名推断 provider**、**自动分组**、**重复账号策略**（跳过/覆盖）
- 如何在导出时增加**头部元信息**与 **GPTMail 分段**，并保持旧文件可导入
- 如何保证**向后兼容**现有单类型导入与导出安全机制（2FA 二次验证）

## 2. 设计原则与硬约束

### 2.1 API 与 DB 硬约束

- **不新增路由**：复用现有端点
  - `POST /api/accounts`：新增 `provider="auto"` 分支触发智能混合导入
  - `GET /api/accounts/export`、`POST /api/accounts/export-selected`：增强导出内容与文件名
- **不改数据库 schema**：`accounts`、`groups`、`temp_emails` 表结构保持不变
- **不改变导出安全机制**：导出仍要求 `X-Export-Token`（2FA 二次验证）

### 2.2 向后兼容原则

- `provider != "auto"` 时：**现有 Outlook/IMAP 导入逻辑保持不变**
- 导入时：**自动跳过 `#` 注释行**（兼容新版导出头部与分段注释）
- 导出时：保留现有行格式（Outlook 4 段、IMAP 3/5 段），仅在顶部新增注释块与末尾新增 GPTMail 分段

### 2.3 GPTMail 的实现约束（与现有架构对齐）

当前系统中 GPTMail 临时邮箱存储在 `temp_emails` / `temp_email_messages` 表，并通过 `/api/temp-emails*` 管理；
因此本 TDD 采用以下设计以满足“导出一个文件→导回来”的闭环：

- **导出**：从 `temp_emails` 读取邮箱地址，写入导出文件的 GPTMail 分段
- **导入（auto 模式）**：遇到 GPTMail 行时写入 `temp_emails`（而非写入 `accounts`）

> 注：FD-00006 中“GPTMail 写入 accounts 表”的表述在当前代码库中会与系统分组（`groups.is_system=1` 的“临时邮箱”）冲突，且会破坏现有 temp_emails 体系，因此在 TDD 中做了实现层面的收敛调整。

## 6. 导入端（provider=auto）详细设计

### 6.1 触发条件与入口改造

文件：`outlook_web/controllers/accounts.py` → `api_add_account()`

当前逻辑：
- `provider != "outlook"` → IMAP 单类型导入（保持不变）
- 否则 → Outlook 单类型导入（保持不变）

新增分支：

```python
provider = (data.get("provider") or "outlook").strip().lower()
if provider == "auto":
    return _handle_auto_import(data)
```

**关键点**：auto 模式允许 `group_id = null`（自动分组），因此不能复用当前“先校验 group_id”的逻辑；需要把 group 校验下沉到 auto 分支内部按规则处理（见 6.4）。

### 6.2 请求字段（provider=auto）

`POST /api/accounts` 请求体（增量字段）：

```json
{
  "account_string": "多行文本…",
  "provider": "auto",
  "group_id": null,
  "duplicate_strategy": "skip",
  "imap_host": "mail.example.com",
  "imap_port": 993
}
```

字段说明：
- `group_id`
  - `null`：自动按 provider 分组（默认）
  - `int`：显式指定分组（所有 Outlook/IMAP 账号统一落该分组；GPTMail 仍写入 `temp_emails`）
- `duplicate_strategy`：`skip`（默认）或 `overwrite`
- `imap_host`/`imap_port`：仅在 “2 段 + 未知域名” 时作为 custom 兜底（可选）

### 6.3 响应结构（增强）

新增 `summary.mode="auto"` 以及按 provider 的统计信息：

```json
{
  "success": true,
  "message": "导入完成：成功 10 个，跳过 2 个，失败 1 个",
  "summary": {
    "mode": "auto",
    "total_lines": 13,
    "imported": 10,
    "skipped": 2,
    "failed": 1,
    "by_provider": {
      "outlook": { "imported": 5, "skipped": 1, "failed": 0 },
      "gmail": { "imported": 3, "skipped": 0, "failed": 0 },
      "custom": { "imported": 1, "skipped": 0, "failed": 0 },
      "gptmail": { "imported": 1, "skipped": 0, "failed": 1 }
    },
    "groups_created": ["Gmail", "自定义IMAP"],
    "errors_total": 1,
    "errors_returned": 1,
    "errors_truncated": false
  },
  "errors": [
    { "line": 12, "email": "expired@gptmail.com", "error": "GPTMail API：邮箱已过期且无法重新注册" }
  ]
}
```

`success` 计算建议（提高可用性）：
- 当 `imported > 0` 或 `skipped > 0` 时，返回 `success=true`（导入流程完成且有有效处理结果）
- 当 `imported==0` 且 `skipped==0`（全部失败或空文件）时，返回 `success=false`

### 6.4 自动分组规则（group_id 处理）

auto 模式分两种：

1) **自动分组**（推荐）：`group_id is None`
- Outlook/IMAP：按 `PROVIDER_GROUP_NAME[provider]` 查找/创建分组并分配
- GPTMail：写入 `temp_emails`，不走 groups（但在 UI/统计中归类到“临时邮箱”）

2) **显式分组**：`group_id is int`
- 校验分组存在且 `is_system=0`（避免导入到系统分组）
- Outlook/IMAP：全部落该分组
- GPTMail：仍写入 `temp_emails`（不依赖 group_id）

### 6.5 类型识别核心：`_detect_line_type`

文件：`outlook_web/controllers/accounts.py`

新增函数（建议定义在 `api_add_account()` 下方，避免污染模块顶层）：

```python
def _detect_line_type(
    line: str,
    fallback_host: str = "",
    fallback_port: int = 993,
) -> Dict[str, Any]:
    """
    返回：
      {
        "type": "outlook" | "imap" | "gptmail" | "error",
        "provider": str,
        "fields": dict,
        "auto_group_name": str,
        "error": str | None,
      }
    """
```

识别优先级（对齐 PRD-00006 §3.1.1 / FD-00006 §4.2）：

1) **空行/注释行**：直接在主循环跳过（不进入该函数）
2) `n >= 5 and parts[2] == "custom"` → 自定义 IMAP（导出格式）
3) `n >= 4` → Outlook（OAuth），refresh_token 合并 `parts[3:]`
4) `n == 3`
   - `parts[2] in KNOWN_PROVIDER_KEYS` → 标准 IMAP（显式 provider）
   - 否则 → error（未知 3 段格式）
5) `n == 2`
   - `infer_provider_from_email(email)` 成功 → 标准 IMAP（域名推断）
   - 否则 → custom IMAP（兜底）
     - `fallback_host` 为空则 error
6) `n == 1`
   - 若是合法邮箱且域名属于 GPTMail（建议白名单，如 `gptmail.com`）→ gptmail
   - 否则 error

字段展开规则：

- Outlook：
  - `fields = {email, password, client_id, refresh_token}`
  - `provider="outlook"`, `auto_group_name="Outlook"`
- IMAP（标准 provider）：
  - `fields = {email, imap_password, imap_host, imap_port}`
  - `provider=<gmail|qq|...|outlook|custom>`
- IMAP（custom）：
  - 5 段：`email----imap_password----custom----host----port`
  - 2 段兜底：`email----imap_password` + (fallback_host/fallback_port)
- GPTMail：
  - `fields = {email}`
  - `provider="gptmail"`, `auto_group_name="临时邮箱"`

微软域名 2 段特例：
- 2 段格式不可能是 OAuth，因此统一按 IMAP 处理（`provider="outlook"`）
- 可在响应 `errors` 或扩展 `warnings` 中提示：“微软邮箱 2 段格式可能需要 OAuth（4 段）”

### 6.6 auto 主流程：`_handle_auto_import`

文件：`outlook_web/controllers/accounts.py`

建议函数签名：

```python
def _handle_auto_import(data: dict) -> Any:
    ...
```

核心流程（伪代码）：

```python
lines = (data.get("account_string") or "").splitlines()
duplicate_strategy = (data.get("duplicate_strategy") or "skip").strip().lower()
fallback_host = (data.get("imap_host") or "").strip()
fallback_port = int(data.get("imap_port") or 993)
explicit_group_id = data.get("group_id")  # None 或 int
use_auto_group = (explicit_group_id is None)

group_cache = {}          # provider -> group_id
groups_created = []

imported = skipped = failed = 0
by_provider = {}
errors = []
errors_total = 0
max_error_details = 50

for line_no, raw in enumerate(lines, 1):
    line = (raw or "").strip()
    if not line or line.lstrip().startswith("#"):
        continue

    result = _detect_line_type(line, fallback_host, fallback_port)
    if result["type"] == "error":
        failed += 1
        errors_total += 1
        append_error(...)
        bump_provider_failed(result.get("provider") or "unknown")
        continue

    provider = result["provider"]
    bump_provider_total(provider)

    if result["type"] in ("outlook", "imap"):
        group_id = explicit_group_id if not use_auto_group else _resolve_auto_group(provider, group_cache, groups_created)
        email = normalize_email(result["fields"]["email"])

        existing = accounts_repo.get_account_by_email(email)
        if existing:
            if duplicate_strategy == "skip":
                skipped += 1
                bump_provider_skipped(provider)
                continue
            if duplicate_strategy == "overwrite":
                _overwrite_account(existing, result, group_id)
                imported += 1
                bump_provider_imported(provider)
                continue

        # 新增账号（复用 accounts_repo.add_account，内部加密）
        add_account_by_type(...)
        imported += 1
        bump_provider_imported(provider)
        continue

    if result["type"] == "gptmail":
        ok = _handle_gptmail_import(...)
        if ok:
            imported += 1
            bump_provider_imported("gptmail")
        else:
            failed += 1
            bump_provider_failed("gptmail")
```

注意事项：
- `accounts_repo.add_account()` 已支持外部事务参数，但 auto 模式建议**逐行提交**以保证“部分成功不回滚”
- 对 `duplicate_strategy` 做白名单校验：非 `skip/overwrite` 直接回退为 `skip`
- 统一邮箱格式校验正则：`^[^@\s]+@[^@\s]+\.[^@\s]+$`

## 7. 重复账号策略与覆盖更新（Repo 改造）

### 7.1 重复策略：skip / overwrite

适用范围：
- 仅 `provider=auto` 时启用（单类型导入保持既有行为：重复视为失败/写入失败）

## 8. GPTMail 导入/导出设计（temp_emails）

### 8.1 GPTMail 导入：`_handle_gptmail_import`

文件：`outlook_web/controllers/accounts.py`

建议新增函数：

```python
def _handle_gptmail_import(email: str, errors: list, line_no: int) -> bool:
    """
    返回 True 表示成功写入 temp_emails；False 表示失败并已写入 errors。
    """
```

处理流程（对齐 PRD-00006 §3.1.6，结合现有 gptmail service 能力）：

1) **重复检查**：`temp_emails_repo.get_temp_email_by_address(email)`
   - 已存在 → 直接返回 True（视为 skipped/imported 由上层统计决定，推荐计入 skipped）
2) **API 可用性检查**：调用 `gptmail.get_temp_emails_from_api(email)`
   - 返回 `list`（可为空）→ 认为邮箱有效
3) **写库**：`temp_emails_repo.add_temp_email(email)`（落 `temp_emails`）
4) 若 API 检查返回 `None`：
   - 尝试“重新注册”：解析 `prefix`/`domain`，调用 `gptmail.generate_temp_email(prefix, domain)`
   - 若返回邮箱与原邮箱一致 → 写库成功
   - 否则 → 失败，写入 errors（包含清晰原因）

限流与滥用防护：
- 单次导入 GPTMail 行数建议限制为 20（超出则记录为失败并提示“超过上限”）
- 对每次 API 调用可插入最小间隔（如 100ms）避免 429

### 8.2 GPTMail 导出

导出来源：`temp_emails_repo.load_temp_emails()`（仅导出邮箱地址）

导出格式（对齐 PRD-00006 §3.2.2）：

```text
# === 临时邮箱（GPTMail）===
tmpuser1@gptmail.com
tmpuser2@gptmail.com
```

> 不导出 `temp_email_messages`（邮件内容），只迁移邮箱地址本身。

## 11. 错误处理、审计日志与边界情况

### 11.1 错误结构与截断

保持与现有导入一致：
- `max_error_details = 50`
- `errors_total` 统计真实失败数，`errors` 只返回前 50 条

建议 errors item 结构：

```json
{ "line": 12, "email": "user@x.com", "error": "错误原因（脱敏）" }
```

### 11.2 常见错误分类（auto 模式）

| 场景 | 处理 |
|---|---|
| 行格式不匹配任何规则 | 该行失败，继续下一行 |
| 3 段 provider 不在 `KNOWN_PROVIDER_KEYS` | 该行失败 |
| 2 段未知域名且无 `fallback_imap_host` | 该行失败 |
| 邮箱正则校验失败 | 该行失败 |
| 覆盖更新时 DB UPDATE 失败 | 该行失败（需写审计/日志） |
| GPTMail API Key 未配置/接口超时 | 该行失败（提示配置/稍后重试） |
| GPTMail 邮箱无法重新注册 | 该行失败（提示“邮箱已过期”） |

## 12. 性能与安全设计

### 12.1 性能

- **自动分组缓存**：`group_cache[provider] -> group_id`，避免每行查库
- **避免不必要的字符串拼接**：导出可使用 `io.StringIO` 追加 lines（账号多时更稳定）
- **GPTMail API 调用限流**：必要时在循环中 sleep（如 100ms）避免 429

### 12.2 安全

- 导出文件包含明文敏感信息（refresh_token/imap_password），但导出操作受 2FA 保护（保持不变）
- 禁止在日志/审计中记录敏感字段原文（password/refresh_token/imap_password）
- 对导入字段使用既有的“控制字符清理 + 长度限制”策略，避免日志污染与异常膨胀

## 13. 兼容性策略

| 场景 | 预期 |
|---|---|
| `provider='outlook'` 旧导入 | 完全不受影响 |
| `provider='gmail/qq/custom...'` 旧导入 | 完全不受影响 |
| `provider='auto'` 导入新版导出文件 | 注释行跳过 + 混合识别成功 |
| `provider='auto'` 导入旧版导出文件（无头部元信息） | 仍可识别（本身就是有效行） |
| 旧版系统导入新版导出文件 | `#` 行被跳过；Outlook/IMAP 有效行可用；GPTMail 1 段行会失败但不影响其他 |
